# 架构说明 — IO 层 / 策略层 边界

本机器人两层之间有**硬边界**。清楚你的改动落在哪一侧, 能省钱。

```
┌─────────────────────────────────────────────────────────────────┐
│  策略层  — 可以自由修改                                          │
│                                                                  │
│  scripts/live_trader.py      生产主循环: 何时开仓 / 仓位 / 退出  │
│  scripts/backtest_*.py       离线回测模拟                        │
│  scripts/sweep_multi.py      多策略参数扫描                      │
│  gridtrader/quant/           指标 / 回测 / 策略 / 存储            │
│  trader/config.py            配置加载                             │
│  trader/models.py            Position 数据结构                    │
│  config/trader.yaml          参数旋钮                             │
│                                                                  │
│         ▲ 只通过 BinanceFutures.<method>(...) 调用               │
└─────────│───────────────────────────────────────────────────────┘
          │
┌─────────│───────────────────────────────────────────────────────┐
│  IO 层  — 视为冻结。仅在 API 迁移时修改。                        │
│                                                                  │
│  trader/exchange.py          BinanceFutures — 所有 REST 调用     │
│  gridtrader/quant/hmac_client.py   签名 + 时钟偏移恢复           │
└──────────────────────────────────────────────────────────────────┘
```

## 为什么要分边界

策略层的 bug 代价是几笔亏损交易。IO 层的 bug 可能:

* 漏挂止损止盈 → 裸仓 → 爆仓
* 循环重试过期订单 → 被交易所封禁
* 发错 `reduceOnly` / `positionSide` → 从对冲变成敞口

IO 层是**承重墙**, 是在 2026-06 重构时有意剥离的。
对它的任何修改, 都要用和 `git push --force` 到 master 同样的谨慎对待。

## 规则

### 策略层改动（>95% 的日常工作）

* 可以: 调整入场阈值、ATR 乘数、盈亏比、熔断上限、仓位管理、
  日志格式, 添加新指标, 添加消费 `BinanceFutures` 的新脚本。
* 涉及文件: `scripts/live_trader.py`, `gridtrader/quant/*`,
  `config/*`, `scripts/*`（只要它们 import `BinanceFutures`）。
* **禁止**在 `trader/exchange.py` 之外的地方写
  `requests.get("https://fapi...")` 或 `hmac.new(...)`。
  如果需要调用 Binance 尚未暴露的接口, 先在单独的 commit 里
  给 `BinanceFutures` 加方法, 并更新合约测试。

### IO 层改动（罕见 — 仅用于 Binance 迁移 / 新接口）

* 必须在**同一个 commit** 里更新 `tests/test_exchange_contract.py`:
  - 新增 public 方法 → 加入 `PUBLIC_METHODS`
  - 新增接口 → 加入 `EXPECTED_ENDPOINTS`
  - 重命名/删除方法 → 审查所有策略层调用方
* PR 描述应注明: "本次改动涉及 IO 合约"。
* 更新文件头部横幅注释的日期。

## 强制执行的测试

```
pytest tests/test_exchange_contract.py    # IO 合约   (5 tests, <1s)
pytest tests/test_indicators.py           # 指标计算   (4 tests, <1s)
pytest tests/test_strategies.py           # 策略信号   (4 tests, <1s)
```

合约测试会在以下情况报错:

* `BinanceFutures` 方法增删但未更新合约;
* 调用了不在白名单上的 Binance 接口;
* 止损止盈意外回退到 `/fapi/v1/order`（自 2025-12-09 起已废弃的路径）;
* 市价单漏传 `reduceOnly`;
* `dry_run=True` 时泄漏了 HTTP 调用。

## Binance USDⓈ-M 合约已知事实（承重级）

* 自 **2025-12-09** 起: STOP_MARKET / TAKE_PROFIT_MARKET /
  TRAILING_STOP_MARKET 必须通过 `POST /fapi/v1/algoOrder`
  + `algoType=CONDITIONAL` + `triggerPrice`（非 `stopPrice`）下单。
  旧接口 `POST /fapi/v1/order` 对这些类型返回 `-4120`。
* 算法订单查询用 `GET /fapi/v1/openAlgoOrders` — 它们**不会**出现在
  `/fapi/v1/openOrders` 里。混淆两者会导致误判仓位为裸仓。
* `closePosition=true` 和 `quantity` 互斥, 二选一。
  `reduceOnly` 不能与 `closePosition=true` 同时发送。
* 时钟偏移 > 1 秒会触发 `-1021 Timestamp ahead`。
  `BinanceFutures` 每 30 分钟自动校时, 且在收到 `-1021` 后下次调用时重同步。
* WSL 无法直连 `fapi.binance.com`。生产环境在 Windows 上运行,
  代理 `http://127.0.0.1:12000`。

## 守规矩的脚本

以下脚本全部 import `BinanceFutures`, 从不自己签名请求:

* `scripts/check_open_orders.py` — 只读持仓盘点
* `scripts/list_algo_orders.py`  — 只读算法订单列表
* `scripts/place_safety_stop.py` — 幂等的止损+止盈挂单
* `scripts/live_trader.py`       — 生产主循环

如果你新增的脚本要和 Binance 通信, 照此办理。
